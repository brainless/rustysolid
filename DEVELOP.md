# DEVELOP

This is a minimal template for fullstack development (human or agent). Shared Rust types drive everything.

## Scope

Only maintain these parts: `backend`, `shared-types`, `gui`, `admin-gui`, `scripts`.
Do not add extra services or crates unless explicitly requested.

## Type-Driven Workflow

1. Define API/domain types in `shared-types/src/*.rs`
2. Regenerate TypeScript types: `cargo run -p shared-types --bin generate_api_types` → `gui/src/types/api.ts`
3. Implement backend handler using shared types
4. Implement UI in `gui`/`admin-gui` against generated types

A feature is complete only when backend + frontend compile against the same shared contract.
Start from `shared-types`, never UI-first. Keep endpoints small and explicit. Prefer strict enums/newtypes over free-text states.

## Project Naming

- Root config: `project.toml` (copy from `project.toml.template`, gitignored)
- Apply names: `scripts/init-project.sh`
- Never hardcode app/repo names in scripts or configs — always parameterize by `project.name`.

## Configuration

Config is resolved in priority order: **env var → `project.toml`** (TOML, typed).

| File | Purpose | Committed |
|---|---|---|
| `project.toml` | Local dev config | No (gitignored) |
| `project.toml.template` | Starter template | Yes |
| `scripts/configs/server.toml` | Prod non-secret config | Yes |
| `server.env` | Server secrets (`DATABASE_URL`) | No (generated by `setup-server.sh`) |

- `project.toml` — local development; read by backend (`config::Config::load()`) and vite apps at dev/build time via `smol-toml`
- `scripts/configs/server.toml` — committed prod config; `deploy.sh` uploads it to `DEPLOY_ROOT/project.toml` on the server
- `server.env` — contains only `DATABASE_URL`; written by `setup-server.sh` to `DEPLOY_ROOT`, permissions `600`; loaded by systemd as `EnvironmentFile=` which overrides the TOML value
- env vars — override any TOML field; applied in `Config::apply_env_overrides()`

Backend helper binaries (`src/bin/`) include `config.rs` via `#[path]` and call `Config::load()`. No manual env setup needed on the server.

Vite apps (`gui`, `admin-gui`) read `project.toml` at build/dev time via `smol-toml` in `vite.config.ts`.

## Structure Rules

- `shared-types` is the source of truth for API payloads
- Avoid handwritten duplicate API types in frontend apps
- Avoid premature abstractions — keep code minimal and typed
